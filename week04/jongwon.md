# 유튜브 동영상 검색

- 동영상 수가 순식간에 늘어나는 플랫폼에서의 동영상 검색 시스템 설계
- 텍스트를 입력하면 그와 관련된 동영상 노출
- 영상의 관련성은 시각적 콘텐츠, 제목 설명 등으로만 결정
- 학습 데이터는 천만 쌍의 <동영상, 텍스트 쿼리>가 있음
- 영어만 지원해도 괜찮음
- 동영상은 10억 개
- 개인화 시스템이 없어도 됨

## 문제 구조화

- 결국 시스템은 텍스트 쿼리를 입력으로 받아 동영상을 관련성에 따라 정렬해서 출력
- 관련성에는 시각적, 텍스트 콘텐츠 모두 활용
- 시각적 검색과 텍스트 검색을 따로 수행 후 합치는 걸 고려해볼 수 있음


- 시각 검색은 텍스트로 영상을 검색함
- 보통 여기에는 표현학습을 사용
- 모델은 동영상을 보고 임베딩 벡터를 생성하는 인코더와 텍스트를 보고 임베딩 벡터를 생성하는 인코더를 지님
- 둘의 유사도 점수는 스칼라 곱을 사용해서 계산


- 텍스트 검색은 보통 역색인을 기반으로 함
- 머신러닝에 기반하지 않아 효율적인 검색이 가능하고, 학습 비용이 들지 않음
- ES 등이 유명하고, Typesense, Clickhouse 등 여러 대체제가 있음

## 데이터 준비

- 거의 모든 머신러닝 알고리즘은 숫자 입력값만 사용
- 텍스트, 동영상 등 비정형 데이터는 수치로 변환해야 함


- 텍스트는 일반적으로 텍스트 정규화, 토큰화, 토큰을 ID로 변환하는 세 단계를 거친다.
    - 정규화
        - `dog`, `DOG`, `DoG`, `DOG!` 등은 사실 다 비슷한 말
        - 대소문자, 문장 부호, 공백은 미리 다듬어 둘 수 있다
        - 그 외에도 NFKD, 악센트 제거, 형태소 분석 및 어간 추출 등이 있다
            - [유니코드 등가성 - 위키백과, 우리 모두의 백과사전](https://ko.wikipedia.org/wiki/%EC%9C%A0%EB%8B%88%EC%BD%94%EB%93%9C_%EB%93%B1%EA%B0%80%EC%84%B1)
            - ﬁ(U+FB01) → f(U+0066) + i(U+0069) 등
    - 토큰화
        - 텍스트 조각을 토큰 단위로 분해하는 절차
        - 단어 토큰화
            - 특정 구분 기호에 따라 텍스트를 개별 단어로 분할 (공백, `'`, `,` 등)
            - 근데 이런 걸 그냥 제거하면 의미가 달라지므로 잘 구분해야 함
        - 서브워드 토큰화
            - n gram 등 하위 단어로 분리
        - 문자 토큰화
            - 텍스트를 문자 집합으로 분할
            - 이건 왜 뭔지 찾아도 잘 안 나오지..
    - 토큰 -> ID 변환
        - 토큰 확보 후에는 토큰을 숫자 ID로 변환해야 함
        - 두 가지 방법이 있음
            - 룩업 테이블
                - 토큰을 고유한 ID에 매핑하고 이를 테이블에 저장
                - 빠르게 변환 가능. ID -> 토큰도 변환 가능
                - 단, 토큰이 많을수록 테이블 사이즈 커짐
                - 미등록된 토큰 대응 불가능
                - 충돌 이슈 없음
            - 해싱
                - 피처 해싱, 해싱 트릭이라고도 부름
                - 룩업 테이블 없이 해시 함수로 ID를 가져오는 방법
                - 매번 해시 함수 계산 필요, ID -> 토큰 변환 불가능
                - 대신, 어떤 토큰이든 변환 가능. 미등록 단어도 처리
                - 충돌 가능성 있음


- 동영상 데이터도 전처리가 필요하다
    - 영상 -> 프레임 쪼개기
    - -> 그 중 샘플링
    - -> 크기 조정해서 맞추기
    - -> 스케일링, 정규화, 색상 모드 보정 등

## 모델 개발

- 텍스트 인코더
- 통계적 기법과 머신러닝 기법이 있음
- 통계적 기법
- BoW(단어 가방 모형, Bag of words)
    - 문장을 fixed length 벡터로 변환
    - 문장을 나타내는 행과 단어 인덱스를 나타내는 열이 있는 행렬을 만들어 문장-단어 발생 행렬을 만든다
        - 문장을 빠르게 표현할 수 있지만, 단점이 있음
            - 단어 순서를 고려하지 않아 다른 문장이지만 의미가 달라질 수도 있음
        - 문장의 의미론적, 문맥적 의미를 파악하지 못함
        - 표현 벡터가 희소함. 크기가 우리가 보유한 고유 토큰 수와 같아서 매우 크고, 대부분은 0으로 채워짐
          ![](files/Pasted%20image%2020260404170753.png)


- TF-IDF(Term Frequency Inverse Document Frequency)
    - 단어가 컬렉션, 말뭉치(corpus)에서 얼마나 중요한지도 반영
    - BoW처럼 행렬을 만들지만, 빈도에 따라 행렬을 정규화
    - 자주 나오는 단어에 가중치를 덜 부여해 BoW보다 더 나은 표현을 제안한다
    - 그러나 아래 한계가 있다
        - 새 문장을 추가하려면 다시 빈도 계산해서 정규화 필요
        - 단어 순서 고려 X
        - 문맥적 의미 포착 X
        - 표현이 희소


- 즉, 통계적 방법은 일반적으로 빠르지만 문맥적 의미를 포착하지 못하고 표현이 희소하다

> 표현이 희소하다

- 벡터 대부분이 `0`이고 실제 값은 일부만 존재
- 차원은 큰데 대부분 의미 없는 값
- 이로 인해 유사도 계산도 어려움
- `[0.12, -0.8, 0.33, ...]` 처럼 대부분의 값이 존재하는 임베딩과 반대되는 지점


- 머신러닝 기법
    - 문장을 임베딩으로 바꾸고, 임베딩 사이의 거리를 단어들의 유사성으로 반영한다
        - 두 단어가 의미적으로 유사하다면 임베딩 공간에서도 가깝다


- 임베딩(룩업) 계층
    - 임베딩 계층으로 각 ID를 임베딩 벡터에 매핑
    - 이를 통해 ID와 같은 희소 피처를 고정된 크기의 임베딩으로 변환 가능
- word2vec
    - 단어 임베딩 생성하는 데 사용하는 모델 제품군
    - 얕은 신경망 아키텍처로 로컬 컨텍스트에서 단어의 동시 출현을 활용하여 단어 임베딩 학습
    - 특히 훈련 단계에서 주변 단어로부터 중심 단어를 예측하는 방법을 학습
        - 특정 단어 주변에 어떤 단어들이 자주 등장하는지 학습해서 의미를 파악
        - "나는 밥을 먹었다", "나는 빵을 먹었다" -> 밥이랑 빵이랑 의미가 비슷하구나
    - 두 가지 주요 모델이 있음. CBOW(Continuous Bag of Words), Skip-gram
        - CBOW
            - 주변 단어 -> 중심 단어
            - 빈칸 채우기처럼 단어를 채워 넣는 방식
            - 흔한 단어 학습 쉽지만 희소 단어를 학습하기엔 어려움
            - 데이터 많으면 안정적임
        - Skip-gram
            - 중심 단어 -> 주변 단어
            - 반대로 '밥'이라는 단어가 나왔으면 주변에 나는, 먹었다 같은 단어가 나오겠다를 학습
            - 희소 단어 잘 학습하고 의미 표현 풍부하지만 느림 (데이터 더 많이 필요)
    - 간단하고 효과적이지만 트랜스포머를 기반으로 한 아키텍처가 더 좋은 결과를 보여주고 있음


- 트랜스포머 기반
    - [Attention Is All You Need](https://arxiv.org/abs/1706.03762)
    - 단어를 임베딩으로 변환할 때 컨텍스트를 고려
    - 문맥에 따라 같은 단어라도 다른 임베딩을 생성
    - 또한 기존의 RNN 등의 순차 처리 한계를 벗어나 병렬 처리도 가능

- 동영상 인코딩
    - 동영상 수준에서 하거나
        - 전체 동영상을 처리해서 임베딩 생성
        - 3차원 합성곱이나 트랜스포머 기반
            - 전체 동영상을 처리하다 보니 속도도 느리고 메모리도 많이 든다
    - 프레임 수준에서 하거나
        - 동영상 및 샘플 프레임 전처리
        - 샘플링된 프레임에서 모델 실행하여 프레임 임베딩 생성
        - 동영상 임베딩을 위해 프레임 임베딩을 집계한다
        - 프레임 수준이라 훨씬 빠르고 계산 비용도 적다

        - 프레임 샘플링은 어떻게 하는가?

            - 혹은 1FPS 등으로 균일하게 뽑거나
            - 장면 전환을 감지하는 모델을 먼저 돌린 후, 장면마다 1~N개의 프레임을 추출하거나
            -
                + near duplicate 감지해서 제거하고

    - 프레임 수준으로 하면 이미 많이 발전한 이미지 모델을 재활용하기 쉬움

    - 영상을 하나의 단위 벡터로 추출하기엔 영상 안에 반대되는 행위들이 포함되면서 정보가 뭉개지기 쉬움. 문 열기 vs 문 닫기, 가까워지거나 멀어지거나 등을 하나의 벡터로 표현하다 보면 정보가

        - 프레임마다 캡션을 달아서 이 정보를 활용해서 시간에 따라서 바뀌는 의미를 드러내거나

        - 등등

    - 한 개의 P, 여러 개의 Negative Query

## 평가

### 오프라인 지표

- `precision@k`, `mAP`
    - `precision@k`는 순위 상위 k 중 관련 항목 수
        - 근데 훈련 세트 positive 데이터가 1개이므로 항상 분자는 최대 1이므로 값이 커질 수가 없다
    - `mAP(mean Average Precision)`
        - mAP는 정답이 여러 개일 때 전체 품질을 보는 것
        - mAP는 순서에 따라서 점수가 크게 달라지는데 검색은 top K가 중요하지 순서가 큰 의미가 없음
        - 반대로 말하면, 유저가 안 보는 낮은 순위의 결과도 점수에 영향을 줌
- `recall@k`
    - 관련 동영상 전체 개수 중 모델이 찾은 개수 비율
    - 관련 동영상 전체 개수는 항상 1이므로, 주어진 텍스트 쿼리로 영상을 K개 내로 찾았는지를 평가하기 좋다
    - 단, k를 잘 선택해야 하며 내가 원하는 영상을 15위로 찾은 거랑 50위로 찾은 건 품질이 다른 지표일 수도 있는데 이런 걸 반영하지는 못한다.
- `MRR(Mean Reciprocal Rank)`
    - 각 쿼리에서 첫 번째 관련 항목 순위의 역수를 평균하기
        - 앞에 나올수록 높은 점수, 뒤로 갈수록 점수에 미치는 영향이 작아짐
        - 순서를 반영하면서 결과에 포함되었는지 여부로 평가 가능

### 온라인 지표

- 클릭률
    - 검색된 동영상 클릭 빈도
    - 클릭이 반드시 검색된 동영상이 관련되었다는 걸 보여주지는 못함. 그냥 재밌어보여서 눌렀을 수도
    - 그래도 간단하게 검색 결과를 클릭한 사람 수를 보여주니 좋은 지표
- 동영상 시청 완료율
    - 검색 결과 중 끝까지 시청한 비율
    - 대신, 이는 일부만 시청했어도 관련된 영상일 수 있는데 이를 반영하지 못함
    - 즉, 시청 완료율만으로는 검색 관련성을 보여주지 못함
- 검색 결과의 총 시청 기간
    - 검색 결과와 관련되었을수록 더 오래 시청하는 경향이 있음

## 서빙

- 예측 파이프라인
    - 시각 검색
        - 텍스트 쿼리 인코딩해서 K-NN으로 유사한 동영상 임베딩 찾기.
    - 텍스트 검색
        - ES 등을 사용해서 텍스트 쿼리와 겹치는 태그, 제목이 있는 영상 찾기
    - 융합 계층
        - 앞 두 단계의 목록을 가져와 새로운 목록으로 융합
        - 쉬운 건 예측된 연관성 점수를 기반으로 동영상 순위를 재조정
        - 더 복잡한 방식은 추가 모델을 채택해서 동영상 순위를 재조정. -> 모델 학습이 필요하고 비용도 더 많이 들고 속도도 느리다
    - 재순위화 서비스
        - 비즈니스 로직, 정책 등을 이용해 순위 조정
