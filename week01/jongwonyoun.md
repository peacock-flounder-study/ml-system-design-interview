## 머신러닝 작업으로 문제를 구조화
- 문제의 구조를 잘 정의해야 함
- 동영상 스트리밍 플랫폼 사용자 참여도를 높여달라?
	- 사용자의 참여를 높이는 건 비즈니스 관점
	- 따라서 이 문제를 머신러닝 작업으로 구조화해야 함
- 현실에서는 주어진 문제 해결에 머신러닝이 필요한지부터 먼저 판단해야 함
- 단, 면접에서는 머신러닝이 도움이 된다고 가정할 수 있으므로 다음 틀을 잡는다
	- 머신러닝 목표 정의
	- 시스템 입출력 정의
	- 적합한 머신러닝 유형 선택

### 머신러닝 목표 정의
- 매출 20% 증가를 목표로 모델을 훈련할 수는 없다
- 비즈니스 목표를 잘 정의된 머신러닝 목표로 정의해야 함

|          애플리케이션          |    비즈니스 목표     |          머신러닝 목표           |
|:------------------------------:|:--------------------:|:--------------------------------:|
|      이벤트 티켓 판매 앱       |   티켓 판매량 증가   |     이벤트 등록 수를 극대화      |
|       비디오 스트리밍 앱       |   사용자 참여 증가   |     사용자 시청 시간 극대화      |
|     광고 클릭 예측 시스템      | 사용자 클릭 수 증가  |          클릭률 극대화           |
| 소셜 미디어의 유해 콘텐츠 감지 |  플랫폼 안정성 증대  | 해당 콘텐츠의 유해성 정확히 예측 |
|        친구 추천 시스템        | 사용자 네트워크 확장 |      형성된 연결 수 극대화       |

### 시스템의 입출력 지정
- 예를 들어 소셜 미디어 플랫폼의 유해 콘텐츠 감지 시스템은 
	- 입력 : 게시물
	- 출력 : 게시물 유해성 여부
- 때에 따라 둘 이상의 머신러닝 모델을 사용할 수도 있음
- 이 경우 입출력 각각
	- 예를 들어 유해 콘텐츠 감지에서
	- 폭력 감지 모델과
	- 노출 감지 모델

### 적합한 머신러닝 유형 선택
- 문제를 머신러닝 구성 방법 여러 개
	- 지도 학습
		- supervised learning 모델은 훈련 데이터세트를 사용해서 작업을 학습
		- 많은 문제가 이 유형에 속함
		- 일반적으로 라벨이 지정된 데이터 세트에서 학습하면 더 나은 결과를 얻을 수 있음
	- 비지도 학습
		- unsupervised learning 모델은 정답이 없는 데이터를 처리하여 예측
		- 데이터 사이에서 의미 있는 패턴을 식별
		- 일반적으로 사용하는 비지도 학습 알고리즘은 클러스터링, 연관, 그리고 차원 축소다
	- 강화 학습
		- 에이전트는 환경과의 상호작용에서 시행착오를 반복하여 수행하는 방법을 찾아낸다
		- 로봇이 방을 돌아다니고, 알파고 바둑 등
- 일반적으로 지도 학습이 비지도, 강화학습보다 자주 쓰이는데 더 학습을 잘하기 때문
- 이 책에서도 대부분 지도 학습

- 지도 학습에도 또 다양한 유형이 있다
	- 회귀 모델
		- 연속적인 숫자값을 예측하는 작업
	- 분류 모델
		- 불연속 클래스 라벨을 예측
		- 이미지가 개, 고양이, 토끼 중 무엇인가?
		- 분류 모델은 두 그룹으로 나뉨
			- 이진 분류 모델
				- 이진 결과를 예측. 이 사진에 개가 포함되어 있는가?
			- 다중 클래스 모델
				- 입력을 둘 이상의 클래스로 분류

## 데이터 준비
- 데이터 엔지니어링과 피처 엔지니어링이라는 프로세스로 어떻게 고품질의 데이터를 모델에 입력할 것인지
### 데이터 엔지니어링
- 데이터의 수집, 저장, 검색 및 처리를 위한 파이프라인을 설계하고 구축하는 일

- 데이터 소스
	- 다양한 소스의 데이터 사용
	- 누가 하고, 얼마나 깨끗하고, 데이터 소스를 신뢰할 수 있는지, 사용자가 생성한 데이터인지 시스템이 생성한 데이터인지
- 데이터 저장소
	- 데이터를 상시로 저장하고 관리하기 위한 데이터베이스
	- 사용 사례별로 서로 다른 DB가 구축되므로 각 DB의 작동 방식을 높은 수준에서 이해하는 게 중요하다
	- 일반적으로 머신러닝 설계 면접에서 DB 상세 질문은 잘 나오지 않는다
- ETL(추출, 변환, 적재)
	- 추출 : 다양한 데이터 소스에서 데이터 추출
	- 변환 : 요구사항에 맞게 데이터 정제, 매핑 및 특정 형식으로 변환
	- 적재 : 변환된 데이터를 파일, 데이터베이스, 데이터 웨어하우스에 적재

- 데이터 유형
	- 크게 정형, 비정형 데이터로 나뉨
	- 정형 데이터는 미리 정의된 데이터 스키마를 따름
	- 비정형 데이터는 그렇지 않음
		- 오디오, 파일, 이미지, 텍스트 등등

- 정형 데이터
	- 수치형 (숫자로 표시되는 모든 데이터)
		- 이산형(1년 판매 주택 수), 연속형(주택 가격)
	- 범주형 (할당된 이름이나 라벨을 기반으로 저장 및 식별)
		- 순위형(미리 결정되거나 순차적인 순서 있는 경우. '싫다', '보통이다', '좋다' 등)
		- 명목형(순서 의미 없는 데이터. 성별)
- 비정형 데이터
	- 오디오, 동영상, 이미지, 텍스트

|      |                정형 데이터                |         비정형 데이터          |
|:----:|:-----------------------------------------:|:------------------------------:|
| 특성 |      사전 정의된 스키마, 검색이 쉬움      |   스키마 없음, 검색이 어려움   |

|  예  | 날짜, 전화번호, 신용카드번호, 주소, 이름  | 텍스트, 오디오, 이미지, 동영상 |

### 피처 엔지니어링(feature engineering)
- 두 가지 프로세스
	- 도메인 지식을 사용하여 원시 데이터에서 예상 피처를 선택하고 추출
	- 예측한 피처를 모델에서 사용할 수 있는 형식으로 변환

- 적절한 피처를 선택하는 게 중요
- 가장 가치 있는 피처를 선택해야 함
- 이 피처 엔지니어링 프로세스에는 주제별 전문 지식이 필요하고 진행 중인 작업에 따라 결과가 많이 달라질 수 있다
- 피처를 예측하여 선택한 후에는 피처 엔지니어링 작업을 통해 적절한 형식으로 변환해야 함

#### 피처 엔지니어링 작업
- 피처 중 일부가 모델에서 사용할 수 없는 형식인 경우가 많다
- 선택한 피처를 모델에서 사용할 수 있는 형식으로 변환
- 누락 데이터 처리, 왜곡된 분포가 있는 값 조정, 범주형 피처 인코딩 포함

- 누락 데이터 처리
	- 운영 환경 데이터에는 값이 누락되는 경우가 많으며 일반적으로 삭제 또는 대체라는 두 가지 방법으로 해결
	- 삭제
		- 모든 피처에서 누락된 값이 있는 레코드를 제거
		- 행 삭제, 열 삭제
		- 단점은 데이터 양이 줄어든다는 것
		- 일반적으로 데이터가 많을수록 더 잘 작동하므로 이상적인 방법은 아님
	- 대체
		- 누락된 값을 특정된 값으로 채우기
			- 기본값으로 채우기
			- 평균, 중앙값 또는 최빈값으로 채우기
		- 단점은 데이터에 노이즈가 생길 수도 있다는 것
	- 완벽한 기법은 없고 장단점이 있음

#### 피처 스케일링
- 피처를 표준 범위와 분포를 갖도록 스케일링하는 프로세스
- 왜 필요한가?
	- 데이터 세트의 피처가 서로 다른 범위에 있는 경우 학습이 어렵다
	- 나이, 소득은 피처 범위가 다를 수 있다
	- 또는 일부 모델은 피처 분포가 왜곡되어 학습이 어렵기도 하다

- 정규화(최대-최소 스케일링)
	- 모든 값이 [0, 1] 범위 내에 있도록 조정
	- 피처의 분포를 변경하지는 않는다
	- 정규 분포를 만드려면 표준화를 사용
$$
Z = \frac{x - x_{\min}}{x_{\max} - x_{\min}}
$$
- 표준화(Z-점수 정규화)
	- 피처의 분포가 평균 0, 표준 편차 1이 되도록 변경
	- $\mu$는 피처의 평균, $\sigma$는 표준 편차
$$
Z = \frac{x - \mu}{\sigma}
$$
- 로그 스케일링
	- 피처의 왜곡을 완화하기 위해서 사용
	- 데이터 분포 왜곡 줄이고, 최적화 알고리즘으로 더 빠르게 수렴 가능
$$
Z = \log(x)
$$

#### 이산화(버키팅)
- 연속형 피처를 범주형 피처로 변환
- 불연속형 버킷으로 나누고, 각 값을 해당 버킷으로 표현
- 이렇게 하면 무한한 수의 가능성을 학습하는 대신 몇 가지 범주 학습에 집중할 수 있음

- 이산화는 불연속형 피처에도 적용 가능
- 범주의 수를 줄일 수 있음

#### 범주형 피처 인코딩
- 대부분 모델의 입출력은 숫자
- 즉, 피처가 범주형인 경우 모델에 보내기 전에 숫자 인코딩 필요
- 정수 인코딩, 원-핫 인코딩, 임베딩 학습 등이 있다

- 정수 인코딩
	- 각각의 고유한 범주에 정수 값을 할당
	- 우수는 1, 양호는 2, 불량은 3 등
	- 이 방법은 정수 값이 서로 순차적인 관계를 갖는 경우 유용
	- 그러나 순서가 없는 경우 정수 인코딩은 좋은 선택이 아니다
- 원-핫 인코딩
	- 각 고유 값에 대한 새로운 이진 피처가 생성된다
	- 색(R,G,B)를 세 가지 이진 피처로 변경 R, G, B 피처에 대해 (1, 0, 0), (0, 1, 0), (0, 0, 1)
- 임베딩 학습
	- 범주형 피처를 N차원 벡터에 매핑
	- 범주형 피처가 취할 수 있는 각 고유 값에 대해 N차원 벡터를 학습하는 프로세스
	- 피처가 갖는 고유 값의 수가 너무 많을 때 유용
		- 이 경우 원-핫 인코딩은 벡터 크기가 너무 커지므로 좋은 옵션이 아니다

	- 어떤 데이터를 사용할 수 있고 어떻게 수집할 수 있는가?
	- 데이터 크기는?
	- 새로운 데이터는 얼마나 자주 입력?
- 데이터 저장소
	- 현재 데이터는 어디에 저장? 클라우드 or 사용자 단말?
	- 데이터를 저장하는 데 적합한 데이터 형식은?
	- 이미지, 데이터 모두 포함될 수 있는 데이터 포인트와 같은 멀티모달 데이터는 어떻게 저장?
- 피처 엔지니어링
	- 원시 데이터를 모델에 유용한 형태로 처리하려면 어떻게?
	- 누락된 데이터는 어떻게 처리?
	- 이 작업에 피처 엔지니어링이 필요한가? 
	- 원시 데이터를 머신러닝 모델에 사용할 수 있는 형식으로 변환하기 위해 어떤 작업을 사용할까?
	- 피처를 정규화해야 할까? 
	- 원시 데이터에서 어떤 피처를 구성해야 할까?
	- 텍스트, 숫자, 이미지 등 서로 다른 유형의 데이터를 어떻게 합칠 계획인가?
- 개인 정보 보호
	- 데이터 민감도는 어느 정도?
	- 사용자가 데이터 프라이버시 우려하고 있는가?
	- 사용자의 데이터 익명화가 필요한가?
	- 사용자 데이터를 서버에 저장할 수 있는가? 혹은 사용자 기기에서만?
- 편향
	- 데이터가 편향되어 있는가?
	- 어떤 종류의 편향이 존재하고, 어떻게 수정?

## 모델 개발

### 모델 선택
- 예측 모델링 문제에 가장 적합한 머신러닝 알고리즘과 아키텍처를 선택하는 프로세스
	- 간단한 기준선 설정
		- 동영상 추천 시스템에서는 가장 인기 있는 동영상을 추천하여 기준선 설정
	- 간단한 모델로 실험
		- 로지스틱 회귀와 같이 빠르게 학습할 수 있는 알고리즘 적용해보면 좋다 
	- 더 복잡한 모델로 전환
		- 만족스럽지 않다면 심층 신경망 등 더 복잡한 모델 고려
	- 더 정확한 예측을 원한다면 여러 모델을 조합
		- 하나의 모델 대신 여러 모델을 조합하면 예측 품질을 높일 수 있다
		- 앙상블을 만드는 방법은 배깅, 부스팅, 스태킹이라는 세 가지 방법으로 수행할 수 있다

- 면접 시 다양한 모델 옵션을 탐색하고 장단점 논의하는 게 중요하다
- 몇 가지 일반적인 모델 옵션은 다음과 같다
	- 로지스틱 회귀
	- 선형 회귀
	- 의사결정 트리
	- 그레디언트 부스트 결정 트리 및 랜덤 포레스트
	- 서포트 벡터 머신
	- 나이브 베이즈
	- FM(Factorization Machines)
	- 신경망

- 다양한 옵션 논의할 때는 알고리즘 간략하게 설명하고 장단점 논의하는 것이 좋다
- 다양한 측면도 고려해야 함
	- 모델이 학습해야 하는 데이터 양
	- 훈련 속도
	- 선택할 하이퍼파라미터와 튜닝 기법
	- 학습 지속 가능성
	- 컴퓨팅 요구사항
	- 모델의 해석 가능성(Interpretability). 더 복잡한 모델은 더 나은 성능을 제공할 수 있지만, 그 결과는 해석하기 어려울 수 있다

- 모든 문제를 해결할 수 있는 만능의 알고리즘은 없으며 다양한 머신러닝 알고리즘과 그 장단점을 잘 이해하고 있는지, 요구사항과 제약 조건에 따라 모델 선택할 수 있는 능력이 있는지 확인하기

### 데이터 세트 구성

- 데이터세트 구성에는 5단계가 있다
	- 원시 데이터 수집
	- 피처 및 라벨 식별
	- 샘플링 전략 선택
	- 데이터 분할
	- 클래스 불균형 해결 (있는 경우)
#### 원시 데이터 수집

- 데이터 준비 단계에서 폭넓게 설명하니 여기서는 반복 X

#### 피처 및 라벨 식별

- 피처는 앞에서 이야기 함
- 데이터 라벨을 만들어보자
- 수동 라벨링, 자동 라벨링 있다

- 수동 라벨링
	- 사람이 데이터에 수작업으로 라벨 지정
	- 사람이 직접 관여하므로 정확하지만 비용이 많이 들고, 느리고, 편견이 개입될 수 있고, 도메인 지식이 필요하며 데이터 프라이버시를 위협할 수 있다
- 자동 라벨링
	- 사람이 달지 않아도 ground truth 라벨이 자동으로 추론된다
		- ground truth : 추론에 의해 제공되는 정보가 아니라 직접 관찰 및 측정에 의해 제공되는 사실
	- 관련성에 따라 뉴스 피드 순위를 매긴다면
		- 사용자의 좋아요 기록을 가지고 확률을 출력하는 모델을 학습하기
		- 이렇게 하면 사람이 라벨 매기지 않아도 라벨 지정 가능

- 이 단계에서는 학습 라벨을 얻는 법과 데이터의 형태를 명확하게 이야기하는 게 중요하다

#### 샘플링 전략 선택
- 모든 데이터를 수집하는 건 어려우므로 샘플링은 데이터 양 줄이는 데 효율적
- convenience 샘플링, snowball 샘플링, stratified 샘플링, reservoir 샘플링, importance 샘플링 등이 있다

[Sampling Methods ](https://www.scribbr.com/methodology/sampling-methods/)

#### 데이터 분할

- 데이터 세트를 학습, 평가(검증), 테스트로 나누는 과정

[Train-Test Split for Evaluating Machine Learning Algorithms](https://machinelearningmastery.com/train-test-split-for-evaluating-machine-learning-algorithms/.)

#### 클래스 불균형 해결

- 클래스 라벨이 왜곡된 데이터 세트를 불균형 데이터 세트라고 함
- 더 많은 비율 -> 다수 클래스
- 더 적은 비율 -> 소수 클래스

- 이는 심각한 문제로 모델에 소수 클래스를 학습하기에 충분한 데이터가 없음을 의미
- 이를 완화하기 위한 다양한 기법이 있다.


- 훈련 데이터 리샘플링
	- 서로 다른 클래스 간의 비율을 조정하여 데이터 균형 맞추기
	- 소수 클래스를 오버샘플링 하거나
	- 다수 클래스를 언더샘플링

![](files/Pasted%20image%2020260205215339.png)


- 손실 함수 적용
	- 클래스 불균형에 견고하게 대응
	- 소수 클래스 데이터 포인트에 더 많은 가중치를 주는 방법
	- 손실 함수의 가중치가 높을수록 소수 클래스에 대해 잘못된 예측을 하면 더 많은 불이익을 준다
	- class-balanced loss, focal loss 등이 사용된다

Class-balanced loss. [https://arxiv.org/pdf/1901.05555.pdf](https://arxiv.org/pdf/1901.05555.pdf).  
Focal loss paper. [https://arxiv.org/pdf/1708.02002.pdf](https://arxiv.org/pdf/1708.02002.pdf).  
Focal loss. [https://medium.com/swlh/focal-loss-an-efficient-way-of-handling-class-imbalance-4855ae1db4cb](https://medium.com/swlh/focal-loss-an-efficient-way-of-handling-class-imbalance-4855ae1db4cb).

#### 손실 함수 선택

- 손실 함수는 모델이 예상 결과를 얼마나 정확하게 예측하는지를 측정하는 지표
- 손실 함수를 사용해 학습 과정에서 최적화 알고리즘이 모델의 매개변수 업데이트해서 손실 최소화 가능하다

- 새로운 손실 함수 설계는 어렵다
- 기존 손실 함수 목록에서 잘 선택만 하면 된다
- 때로는 문제에 맞게 약간 변경하는 정도?

#### 처음부터 훈련하기와 미세 조정하기

- 미세 조정은 학습된 파라미터를 약간씩 변경해서 새로운 데이터로 모델 계속 학습하는 것
- 이건 면접관과 논의할 수 있는 설계 결정 사항

#### 분산 학습

- 데이터 많아지고 모델이 커지면서 대규모 학습이 점점 중요해짐
- 분산 학습(distributed training)은 여러 작업자 노드에 작업 분할해서 모델 훈련
- 속도 높이기 위해서 병렬 작동
- 데이터 병렬 처리, 모델 병렬 처리 두 가지 유형 있음

[Understanding Data Parallelism in Machine Learning](https://www.telesens.co/2017/12/25/understanding-data-parallelism-in-machine-learning/)

[Introduction to Model Parallelism](https://docs.aws.amazon.com/sagemaker/latest/dg/model-parallel-intro.html)

- 과제에 따라 분산 학습이 필요할 수도 있는데 면접관과 논의하는 게 중요하다
- 분산 학습은 특정 과제와 관계없이 이야기할 수 있는 일반적인 주제라는 걸 기억하자

#### 논의 주제 
- 모델 선택
	- 적합한 모델은 뭐고, 모델의 장단점은? 다음을 고려해야 한다
		- 훈련에 걸리는 시간
		- 모델이 예상하는 학습 데이터의 양
		- 모델에 필요할 수 있는 컴퓨팅 자원
		- 추론 시 모델의 처리 시간
		- 모델을 사용자 단말에 배포 가능한가?
		- 모델의 해석 가능성(interpretability). 복잡하게 만들면 성능이 올라가도 결과 해석이 어려워질 수 있다
		- 연속학습(continual training) 활용 가능? 아니면 처음부터 다시 훈련?
		- 모델에 몇 개의 파라미터? 얼마나 많은 메모리 필요?
		- 신경망인 경우 ResNet 또는 Transformer 기반 아키텍처 등 일반적인 아키텍처/블록 등에 대해 논의할 수 있다. 은닉 계층의 수, 뉴런 수, 활성화 함수 등 하이퍼파라미터 선택도 논의할 수 있음
- 데이터세트 라벨
	- 라벨은 어떻게 얻을 수 있나?
	- 데이터에 주석을 달았다면 주석의 수준은 어느 정도?
	- 자동 라벨을 사용할 수 있다면 어떻게 해야 하나?
	- 시스템에 대한 사용자 피드백은 어떻게 받나?
	- 자동 라벨을 얻는 데 얼마나 걸리나?
- 모델 훈련
	- 어떤 손실 함수를 선택? (cross entropy, 평균 제곱 오차(MSE), 평균 절대 오차(MAE), Huber Loss 등)
	- 어떤 정규화 사용? (L1, L2, Entropy Regularization, K-fold CV, dropout)
	- 역전파는 무엇인가?
	- SGD, AdaGrad, Momentum, RMSprop과 같은 일반적인 최적화 방안을 설명해야 할 수도 있다
	- 어떤 활성화 함수를 사용하고자 하고, 그 이유는? (ELU, ReLU, Tanh, Sigmoid)
	- 불균형한 데이터 세트 처리 방법은?
	- 편향-분산 트레이드오프는 무엇인가?
	- 과대적합과 과소적합의 원인은 무엇인가? 어떻게 해결해야 할까?
- 연속학습
	- 새로운 데이터 포인트가 나올 때마다 온라인으로 모델을 훈련시키고 싶나?
	- 모델을 각 사용자에게 맞게 개인화해야 하는가?
	- 모델을 얼마나 자주 훈련시켜야 하나?
		- 어떤 모델은 매일 또는 매주 훈련시켜야 하고, 어떤 모델은 매월 또는 매년 훈련시켜야 한다.

#### 오프라인 평가

- 모델 개발 단계에서 성능을 평가
- 일반적으로 평가 데이터세트를 사용하여 예측
- 오프라인 지표를 사용하여 예측이 ground truth에 얼마나 가까운지 측정

| 작업     | 오프라인 지표                                                                            |
| ------ | ---------------------------------------------------------------------------------- |
| 분류     | precision(정밀도), recall(재현율), F1 score, accuracy, ROC-AUC, PR-AUC, confusion matrix |
| 회귀     | MSE, MAE, RMSE                                                                     |
| 순위     | precision@k, recall@k, MRR, mAP, nDCG                                              |
| 이미지 생성 | FID, inception score                                                               |
| 자연어 처리 | BLEU, METEOR, ROUGE, CIDEr, SPICE                                                  |

- 면접 중에는 오프라인 평가에 적합한 지표를 파악하는 게 중요하다
- 쉬운 문제를 해결하려는 경우 순위 지표와 그 장단점을 논의할 수 있어야 함

#### 온라인 평가

- 배포 후 모델이 운영 환경에서 어떻게 수행하는지 평가하는 프로세스
- 모델의 영향을 측정하려면 다양한 지표 정의가 필요
- 보통 비즈니스 목표와 관련되어 있다

| 문제        | 온라인 지표                       |
| --------- | ---------------------------- |
| 광고 클릭 예측  | 클릭률, 수익 증가 등                 |
| 유해 콘텐츠 감지 | 전파율, 유효한 이의 제기 등             |
| 영상 추천     | 클릭률, 총 시청 시간, 시청 완료된 영상 수 등  |
| 친구 추천     | 하루에 전송되는 요청 수, 일일 요청 수락 건수 등 |

- 일반적으로 많은 기업에서 온라인 지표를 추적
- 면접에서는 가장 중요한 몇 가지 지표 선택이 필요
- 온라인 지표 또한 주관적이므로 제품 담당자와 비즈니스 이해관계자에 따라 달라진다

- 이 단계에서는 면접관이 지원자의 비즈니스 감각을 평가
- 따라서 지원자는 자신의 사고 과정을 전달하고 특정 지표를 선택한 이유를 설명하는 게 좋다

#### 논의 주제

- 온라인 지표
	- 효과를 측정하는 데 중요한 지표는?
	- 이러한 지표는 비즈니스 목표와 어떤 연관?
- 오프라인 지표
	- 모델의 예측 능력을 평가하는 데 적합한 오프라인 지표는?
- 공정성 및 편향성
	- 연령, 성별, 인종 등 다양한 대상에 대해 편견을 가질 가능성이 있나?
	- 이 문제를 어떻게 해결할 수 있나?
	- 악의적인 의도가 있는 사람이 시스템에 접근하면?

### 배포 및 서비스 제공

#### 클라우드와 온디바이스 배포 비교

| 항목 | 클라우드 | 온디바이스 |
|---|---|---|
| 단순성 | ✓ 클라우드 기반 서비스를 사용하여 배포 및 관리가 간편함 | ✗ 디바이스에 모델을 배포하기는 간단하지 않음 |
| 비용 | ✗ 클라우드 비용이 높을 수 있음 | ✓ 디바이스 내에서 계산을 수행하면 클라우드 비용 없음 |
| 네트워크 지연 시간 | ✗ 네트워크 대기 시간이 존재함 | ✓ 네트워크 지연 시간 없음 |
| 추론 처리 시간 | ✓ 일반적으로 더 강력한 기계로 인해 추론 속도가 빨라짐 | ✗ 머신러닝 모델 실행 속도 저하 |
| 하드웨어 제약 조건 | ✓ 제약 조건 별로 없음 | ✗ 제한된 메모리, 배터리 소모 등 더 많은 제약이 있음 |
| 개인 정보 보호 | ✗ 사용자 데이터가 클라우드로 전송되므로 개인정보 보호 낮음 | ✓ 데이터가 디바이스를 벗어나지 않으므로 개인 정보 보호 강화 |
| 인터넷 연결에 대한 의존성 | ✗ 클라우드로 데이터를 송수신하는 데 필요한 인터넷 연결 | ✓ 인터넷 연결 필요 없음 |

#### 모델 압축

- 모델을 더 작게 만드는 것
- 추론 처리 시간과 모델 크기 줄이는 데 필요

- 지식 증류(knowledge distillation)
	- 작은 모델(student model)이 큰 모델(teacher model)을 모방하도록 훈련
- 가지치기(pruning)
	- 가장 유용성이 낮은 매개변수를 찾아서 0으로 설정
	- 더 적은 수의 모델을 더 효율적으로 저장 가능
- 양자화(quantization)
	- 모델 파라미터는 32비트 부동 소수점으로 표현되는 경우가 많다
	- 양자화에서는 더 작은 비트를 사용해 파라미터를 표현해 모델 크기가 줄어든다
	- 양자화는 훈련 중, 또는 훈련 후에도 발생할 수 있다

[Quantization](https://pytorch.org/docs/stable/quantization.html)

[Model compression survey](https://arxiv.org/pdf/1710.09282.pdf.)

### 운영 환경에서의 테스트

- 모델이 운영 환경에서 잘 작동하는지 확인하는 유일한 방법은 실제 트래픽으로 테스트하기
- 모델을 테스트하는 데 일반적으로 사용되는 기법으론 shadow deployment, A/B 배포, 카나리 배포, 인터리빙 실험(interleaving experiments), 밴딧(bandits) 등

#### 섀도 배포

- 기존 모델과 병렬 배포
- 각 수신 요청은 두 모델에 라우팅. 기존 모델의 예측만 사용자에게 제공됨

- 새로 개발된 모델의 테스트가 완료될 때까지 신뢰할 수 없는 예측을 할 위험을 최소화할 수 있음
- 그러나 이는 예측 횟수를 두 배로 늘려 비용이 많이 든다

#### A/B 배포

- 새 모델을 기존 모델과 병렬 배포
- 트래픽 일부는 새 모델에게 라우팅

- 두 가지 중요한 요소
	- 각 모델로 라우팅되는 트래픽은 무작위여야 함
	- 충분한 수의 데이터 포인트에 대해 실행해야 적절한 결과를 얻을 수 있음

### 예측 파이프라인

- 운영 환경에서 요청 처리하려면 예측 파이프라인 필요
- 온라인 예측, 배치 예측 중 하나 선택하는 건 설계에서 중요한 결정
#### 배치 예측

- 모델이 주기적으로 예측 수행
- 미리 계산되므로 모델이 예측 생성하는 데 걸리는 시간을 걱정할 필요가 없다
- 그러나 두 가지 단점
	- 사용자의 선호도 변화에 대한 모델의 반응성이 떨어진다
	- 사전 계산해야 할 내용을 미리 알고 있을 때만 가능

#### 온라인 예측

- 요청이 도착하는 즉시 예측을 만들어서 반환
- 모델이 예측을 생성하는 데 너무 오래 걸릴 수 있다

- 배치 vs 온라인은 주로 제품 요구사항에 따라 결정된다
- 온라인은 미리 계산할 수 없는 경우
- 배치는 시스템이 대량의 데이터를 처리하고, 그 결과가 실시간으로 필요하지 않은 경우 이상적

- 머신러닝 시스템 개발에는 단순한 머신러닝 모델링 이상이 필요하다
- 전반적인 머신러닝 시스템 설계를 제안하기 위해서는 다양한 구성요소가 전체적으로 어떻게 함께 작동하는지에 대한 깊은 이해가 필요하다
	- 면접관은 이를 중요한 능력으로 평가

### 모니터링

- 모니터링은 다양한 평가 지표를 추적, 측정 및 기록하는 작업
- 운영 중인 머신러닝 시스템은 여러 이유로 장애가 발생할 수 있다
- 모니터링은 시스템 장애가 발생하면 이를 감지하여 가능하면 빨리 해결할 수 있도록 도와준다
- 면접에서 논의해야 할 중요한 주제이며, 두 영역이 있음

#### 운영 환경에서 시스템에 장애가 발생한 경우

- 보통 데이터 분포 변화로 장애가 발생한다.
- 이는 실제 업무에서 접하는 데이터가 학습 중에 접한 데이터와 다른 경우
- 현실 데이터 분포는 끊임없이 변한다. 즉, 학습에 사용되는 데이터는 시간이 지날수록 관련성이 떨어질 가능성이 높다
- 시간이 지남에 따라 성능이 저하되는 오래된 모델이 된다는 뜻

- 이 문제를 감지하기 위해서는 시스템을 지속적으로 모니터링

- 데이터 분포 변화를 처리하는 방법은 크게 두 가지
	- 대규모 데이터 세트를 학습
		- 충분히 큰 훈련 데이터세트를 사용하면 모델이 더 넓은 분포를 학습할 수 있으므로, 운영 환경에서 발생하는 모든 데이터 포인트는 이 학습된 분포에 포함될 가능성이 높다
	- 새로운 분포의 라벨이 지정된 데이터를 사용하여 모델을 정기적으로 재학습

#### 모니터링 대상

- 운영 환경 구축 후 모니터링 기술에 중점
- 우리의 목표는 머신러닝 시스템 장애를 감지하고 변화를 파악
- 크게 운영 관련 지표와 머신러닝 관련 지표의 두 가지 버킷으로 분류 가능

- 운영 관련 지표
	- 시스템이 정상적으로 작동하는지
	- 평균 서빙 시간, 처리량, 예측 요청 수, CPU/GPU 사용률

- 머신러닝 관련 지표
	- 입출력 모니터링 : 사용하는 데이터만큼 성능이 향상되므로 입출력 모니터링이 중요
	- 드리프트 : 입출력을 모니터링하여 기본 분포 변화 감지
	- 모델 정확도 : 정확도가 특정 범위 내에 있다고 예상하기
	- 모델 버전 : 배포된 모델 버전 모니터링
