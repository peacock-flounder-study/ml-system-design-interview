# 6장 동영상 추천 시스템

- 보고있는 동영상과 유사 동영상 추천? or 메인에 뜨는 개인화된 추천?
	- 후자
- 글로벌 서비스 -> 사용자는 전세계에 있고, 동영상에 쓰이는 언어가 다름
- 동영상에 대한 상호작용으로 데이터 세트 구성
- 재생 목록 만들어서 동영상 그룹화 가능? 재생 목록을 데이터로 활용 가능?

## 머신러닝 작업으로 문제 구조화
### 머신러닝 목표 정의
- 비즈니스 목표는 참여도 증가
	- 사용자 클릭 수 극대화
		- 큰 단점이 하나 있음
		- 제목, 썸네일이 매력적이지만 실제 콘텐츠는 별로인 낚시성 동영상 추천할 수도 있음
	- 시청 완료 동영상 수 극대화
		- 사용자가 끝까지 시청할 가능성이 있는 영상 추천
		- 단, 그러기 쉬운 짧은 영상을 추천하기 쉬워짐
	- 총시청 시간 극대화
		- 영상 시청 시간 자체를 극대화
	- 관련 동영상 수 극대화
		- 엔지니어 혹은 제품 관리자가 몇 가지 규칙에 따라 관련성 정의
		- 암시적, 명시적 사용자 반응을 기반으로 정의
			- 좋아요 누름
			- 절반 이상 시청함 등등
		- 관련성 정의 후에는 데이터 세트 구성 후 모델 훈련하여 연관성 점수 예측 가능

- 여기서는 마지막 관련 동영상 수 극대화 택함
	- 입출력 자유도 높고
	- 단점도 없음

### 시스템의 입력 및 출력 지정
- 사용자를 입력으로 받아 관련성 점수에 따라 정렬된 순위가 매겨진 동영상 목록 출력

### 적합한 머신러닝 유형 선택
- 3가지 유형의 개인화 추천 시스템
	- 콘텐츠 기반 필터링
		- 동여상 피처 사용해서 과거에 관련성 있다고 판단한 동영상과 유사한 동영상 추천
		- 장점
			- 새로운 동여상 추천하기 위해서 사용자 데이터가 필요없음
			- 사용자의 고유한 관심사 파악 가능
		- 단점
			- 새로운 관심사 발견이 어려움
			- 도메인 지식이 필요
				- 수동으로 동영상 피처를 엔지니어링 해야할 때가 많음
	- 협업 필터링
		- collaborative filtering, CF은 사용자 간 유사성 또는 동영상들 간의 유사성 사용하여 새로운 동영상 추천
		- 협업 필터링은 비슷한 사용자가 비슷한 동영상에 관심이 있다는 아이디어에 기반
		- 장점
			- 동영상 피처에 의존하지 않아 도메인 지식 필요 X
			- 사용자의 새로운 관심 분야 쉽게 발견
			- 효율적
				- 동영상 피처에 의존하지 않아 빠르고 자원 소모도 적음
		- 단점
			- 콜드 스타트 문제
				- 새로운 동영상 또는 사용자에 관해 제한된 데이터만 사용할 수 있어 정확히 추천할 수 없는 상황
				- 새로운 사용자, 동영상에 대한 과거 상호작용 데이터가 부족해서 그럼
			- 특이한 관심사 처리 불가능
				- 전문적이거나 특이한 관심사 가진 사용자 대응이 어렵다
	- 하이브리드 필터링
		- 위 두 필터링 모두 사용
		- 순차적 or 병렬로 결합
		- 대부분의 기업이 순차적 하이브리드 필터링 사용
		- 사용자의 데이터 + 동영상 데이터 사용해서 더 정확하게 추천 가능

- 여기서도 하이브리드 필터링을 사용한다
- 구글 논문에서는 후보 생성기로 협업 필터링 기반을 사용하고, 두 번째 단계로 콘텐츠 기반 모델을 사용한다
- [Deep Neural Networks for YouTube Recommendations](https://static.googleusercontent.com/media/research.google.com/ko//pubs/archive/45530.pdf)

## 데이터 준비
### 데이터 엔지니어링
- 동영상
	- 동영상 파일, ID, 길이, 제목, 메타데이터 등
	- 일부는 명시적으로 등록. 일부는 시스템이 암시적으로 파악
- 사용자
	- 사용자 ID, 이름, 나이, 성별, 도시, 언어, Time zone 등
- 사용자-동영상 상호작용
	- 좋아요, 클릭, 노출 수, 과거 검색 등등
	- 위치 + timestamp 등 기타 컨텍스트 정보와 함께 기록됨

### 피처 엔지니어링
- 동영상 피처
	- 동영상 ID
		- 범주형 데이터
		- 숫자 벡터로 표현하기 위해 임베딩 계층 사용
		- 임베딩 학습은 모델 학습 중 진행
	- 동영상 길이
		- 얼마나 긴지 대략 정의
		- 사용자에 따라 선호하는 길이가 다름
	- 언어
		- 사용자는 특정 언어를 선호
		- 범주형 변수, 정해진 범위의 이산값 집합을 가진다
		- 따라서 임베딩 계층으로 표현
	- 제목 및 태그
		- 동영상 설명에 사용
		- 수동으로 제공 or 모델이 암시적으로 예측
		- 제목과 태그는 중요한 예측 변수
		- 태그는 CBZOW 같은 경량 사전 학습 모델로 피처 벡터에 맵핑
- 사용자 피처
	- 사용자 개인정보
		- ID -> 임베딩
		- 나이 -> 버킷화 + one-hot
		- 성별 -> one-hot
		- 언어, 도시, 국가 -> 임베딩
	- 상황별 정보
		- 시간대
			- 시간대에 따라 다른 영상을 볼 수도 있음
			- 업무중 vs 여가 중
		- 단말
			- 모바일 단말에서 짧은 동영상 선호할 수도 있음
		- 요일
			- 요일에 따라 선호 동영상이 다를 수 있음
- 사용자 기록 상호작용
	- 검색 기록
		- 사용자의 과거 행동은 미래 예측에 도움이 됨
		- BERT처럼 사전 학습된 단어 임베딩 모델로 검색 쿼리를 임베딩 벡터에 맵핑
		- 검색 기록은 가변 길이의 텍스트 목록이므로 모든 검색 쿼리를 요약하는 고정 크기의 피처 벡터 생성을 위해 쿼리 임베딩 평균 구함
	- 좋아요 누른 동영상
		- 좋아요는 관심 동영상 파악에 도움이 됨
		- 동영상 ID를 임베딩하고, 이를 평균화하여 좋아요 표시된 동영상의 고정 크기 벡터 얻는다
	- 시청한 동영상 및 노출 수
		- 좋아요랑 비슷

## 모델 개발
### 행렬 인수분해(matrix factorization)
![](files/Pasted%20image%2020260422210420.png)
- 피드백 행렬
	- 유틸리티 행렬
	- <사용자, 동영상> 쌍을 관찰됨 또느 포지티브로 지칭
	- 피드백에는 3가지 옵션
		- 명시적 피드백
			- 좋아요 등 명시적인 상호작용 기반
			- 의견 정확하게 반영하지만 행렬의 희소하다
			- 훈련이 어렵다
		- 암시적 피드백
			- 클릭 시간, 시청 시간 등 사용자의 의견을 암시적으로 나타내는 상호작용 사용
			- 더 많은 데이터 얻지만 노이즈 많을 수 있음
		- 명시적 및 암시적 피드백 조합
			- 휴리스틱 사용하여 위를 조합
	- 가장 적합한 건?
		- 여기선 관련성 극대화이므로 명시적 + 암시적 피드백 결합한다
- 행렬 인수분해 모델
	- 행렬 인수분해는 간단한 임베딩 모델
	- 피드백 행렬을 두 개의 저차원 행렬 곱으로 분해
	- 사용자 임베딩 X 동영상 임베딩
	- 즉, 모델은 각 사용자와 각 동영상을 임베딩 벡터로 맵핑하여 거리가 관련성을 나타내도록 학습
- 행렬 인수분해 훈련
	- 두 임베딩 행렬 생성하여 둘의 곱이 피드백 행렬의 근사치가 되도록
	- 두 개의 임베딩 행렬을 무작위 초기화 후 예상 점수 행렬, 피드백 행렬 사이의 손실을 줄이기 위해 임베딩을 반복적으로 최적화
	- 손실 함수는?
		- 관찰된 <사용자, 동영상> 쌍의 제곱 거리
			- 관찰된 쌍만 보면 관찰되지 않은 쌍에 대해 불이익을 주지 않아 임베딩이 잘못될 수 있다
		- 관찰된, 관찰되지 않은 <사용자, 동영상> 쌍 모두에 대한 제곱 거리
			- 위 문제는 해결
			- 그러나 피드백 행렬은 희소하기 때문에 관찰되지 않은 쌍이 많아 예측값이 대부분 0에 가까워진다
		- 관찰된 쌍과 관찰되지 않은 쌍에 대한 제곱 거리의 가중치 조합
			- 가중치를 부여
			- 훈련 단계에서 한 쪽이 다른 쪽을 지배하지 않도록 보장
			- 이게 좋다
- 행렬 인수분해 최적화
	- SGD(Stochastic Gradient Descent) : 손실 최소화를 위해 사용하는 최적화 알고리즘
		- 전체 데이터(Batch) 대신 일부 데이터의 모음(Mini-Batch)를 사용하여 Loss Function을 계산한다. 
		- Batch Gradient Descent보다 다소 부정확할 수는 있지만, 계산 속도가 훨씬 빠르기 때문에 같은 시간에 더 많은 step을 갈 수 있으며, 여러 번 반복할 경우 Batch 처리한 결과로 수렴한다.
	- WALS(Weighted Alternating Least Squares) : 행렬 인수분해 특화
		- 하나의 임베딩 매트릭스(U)를 고정하고 다른 임베딩(V)를 최적화
		- 다른 임베딩 행렬(V)를 고정하고 임베딩 행렬(U)를 최적화
		- 반복
		- 일반적으로 더 빠르게 수렴하고 병렬화
- 행렬 인수분해 추론
	- 임의의 사용자와 후보 동영상 간의 관련성 예측을 위해 스칼라곱과 같은 유사성 측정값을 사용하여 임베딩 간의 유사성 계산
- 장점
	- 행렬 인수분해는 효율적이므로 학습 속도가 빠르다
	- 학습된 임베딩은 정적이라 일단 학습하면 변환 없이 재사용 가능
- 단점
	- 사용자, 동영상 상호작용에 의존. 연령, 언어 등 다른 피처는 사용하지 않음
	- 신규 사용자는 상호작용이 부족해 추론이 어려움

### 투-타워 신경망 (two-tower neural network)
- 사용자 타워, 동영상 타워라는 두 개의 인코더 타워로 구성
- 사용자 인코더는 사용자 피처를 입력을 받아 임베딩
- 동영상 인코더는 동영상 피처를 받아 임베딩
- 공유 임베딩 공간에서 임베딩 사이의 거리는 관련성을 나타낸다
- 행렬과 다르게 모든 종류의 피처 통합 가능

#### 데이터 세트 구성
- 서로 다른 <사용자, 동영상> 쌍에서 피처를 추출하고 사용자의 피드백에 따라 포지티브 또는 네거티브로 라벨 지정하여 데이터 세트 구성
	- 좋아요 누르거나 절반 이상 시청하면 포지티브
	- 관련성이 없는 동영상 무작위로 선정하거나 싫어요 버튼 눌렀으면 네거티브 라벨
- 사용자는 일반적으로 관련성있는 극히 일부 영상만 찾는다
- 따라서 포지티브 쌍보다 네거티브 쌍이 더 많은 불균형한 데이터 세트가 만들어짐
- 앞에서 얘기한 것들로 해결

#### 손실 함수 선택
- 이진 라벨 예측이므로 분류 작업
- 크로스 엔트로피 등 일반적인 손실 함수 사용
#### 투-타워 신경망 추론
- 임베딩 사용해서 가장 관련성 높은 영상 찾는다
- 즉, 전형적인 최근접 이웃 문제

- 투-타워 신경망은 콘텐츠 기반 필터링과 협업 필터링 모두에 사용할 수 있다
- 협업 필터링에 사용하려면 동영상 ID를 임베딩 벡터로 변환해서 사용
	- 다른 피처 사용 X

- 장점
	- 사용자 피처 잘 활용
	- 신규 사용자 처리 가능
- 단점
	- 쿼리 시점에 사용자 임베딩 계산하므로 서빙 속도가 느려진다
	- 콘텐츠 기반 필터링에 모델을 쓰면 동영상 피처를 동영상 임베딩으로 변환해야 하므로 추론 시간이 늘어난다
	- 학습 매개변수가 많아서 더 많은 컴퓨팅 자원이 필요하다

- 다양성
	- 추천 동영상이 서로 얼마나 다른지 측정
	- 사용자가 다양한 동영상에 관심이 있으므로 이 지표는 반드시 추적
	- 다양성 측정을 위해 목록에 있는 동영상 간의 평균 쌍별 유사도 (코사인 유사도, 스칼라 곱 등)을 계산
	- 유사도 점수가 낮으면 다양하다는 뜻

- 다양성을 유일한 척도로 사용하면 오히려 잘못된 해석이 나올 수도 있다
- 추천 동영상이 다양하지만 사용자와 관련없는 케이스를 구분할 수 없음

### 온라인 지표
- 클릭률(CTR)
	- 클릭된 동영상과 총 추천 동여상 비율
	- 사용자 참여 추적하는 유용한 지표지만 낚시성 동영상을 거르거나 측정할 수 없다
- 시청 완료 동영상 수
	- 끝가지 시청한 추천 동영상ㄴ 개수
- 총 시청 시간
	- 추천 동영상 시청한 총시간
- 명시적인 사용자 피드백
	- 명시적으로 좋아요, 싫어요 표시한 개수

## 서빙
- 서빙 시점에 시스템은 수십억 개의 동영상 중에서 선택 범위를 좁혀 특정 사용자에게 가장 관련성이 높은 동영상 추천
- 수십억 개의 동영상 사용할 수 있는 상황에서 많은 피처를 입력으로 사용하는 무거운 모델을 선택하면 서빙 속도가 느려질 수 있다
- 반면에 경량 모델은 관련성이 높은 동영상 추천이 어렵다
- 다단계 설계에서 두 개 이상의 모델 사용이 자연스럽다
- 경량 모델은 후보를 생성하여 빠르게 동영상 범위를 좁히고, 두 번째 단게에서는 동영상에 정확한 점수를 매기고 순위를 매기는 더 무거운 모델을 쓴다

### 후보 동영상 생성
- 수십억 개를 수천 개로 좁히는 게 목표
- 정확성보다 효율성을 우선시해 오탐 걱정 X

- 후보 모델 빠르게 생성하기 위해 동영상 피처에 의존 X
- 새로운 사용자를 처리할 수 있어야하므로 투-타워 신경망 적합

- 사용자 임베딩 가져와서 근접 동영상 검색 -> 순위 매기고 출력 반환
- 성능 올리기 위해서 기업들은 한 번 이상의 후보 동영상 생성 단계를 사용할 수 있다
	- 여러 가지 이유로 동영상에 관심 가질 수 있음
	- 인기 있거나, 유행하거나, 자신의 위치와 관련있거나 등등
	- 그래서 한 개 이상의 후보 동영상 생성을 사용
	- 수십억 개에서 수천 개로 범위를 좁히면 점수 매기기

### 점수 매기기
- 사용자 및 후보 동영상 입력 받아 점수 매기고 순위 매겨진 동영상 목록 출력
- 이 단계에서는 효율성보다 정확성 우선시
- 컨텐츠 기반 필터링 선택하고, 동영상 피처에 의존하는 모델 사용
	- 영상 수가 적어 더 무거운 모델 가능

### 재순위화
- 기준이나 제약 조건 추가해서 동영상 순위 재조정
- 동영상이 낚시성인지 파악하거나 등등
- 아래 정도
	- 지역 제한 동영상
	- 최신 동영상
	- 잘못된 정보를 퍼뜨리는 영상
	- 중복 또는 거의 중복된 동영상
	- 공정성 및 편향
### 동영상 추천 시스템의 과제
#### 추천 속도
- 신속함이 매우 중요
- 그러나 개수가 많으면 효율적이고 정확하게 추천이 어려움
- 그래서 2단계 설계 사용
- 유튜브도 비슷한 방식, 인스타도 2단계 사용
	- [Powered by AI: Instagram’s Explore recommender system \| Instagram Engineering](https://instagram-engineering.com/powered-by-ai-instagrams-explore-recommender-system-7ca901d2a882)
#### 정밀도
- 정밀도 보장 위해서 X는 동영상 피처는 물론 더 많은 피처를 사용하는 강력한 모델로 동영상 순위 매겼음

#### 다양성
- 사용자는 다양한 동영상 포함되기를 원함
- 여러 후보 생성기 사용

#### 콜드 스타트 문제
- 신규 사용자
	- 어떠한 상호작용 데이터도 없음
	- 이 경우 나이, 성별, 언어, 위치 등의 피처를 기반으로 투-타워 신경망 사용하여 예측
	- 그래서 어느 정도 개인화된 영상 추천 가능
- 새 동영상
	- 동영상 메타데이터와 콘텐츠 사용할 수 있지만 상호작용은 없음
	- 이를 처리하는 한 가지 방법은 휴리스틱
	- 무작위 사용자에게 동영상 표시하고 상호작용 데이터를 수집
	- 상호작용 충분히 수집하면 새로운 상호작용을 사용하여 신경망 조정 가능

#### 훈련 확장성
- 대규모 데이터세트 쓰면 비용 문제가 있음
- 새로운 상호작용을 지속해 추가하며, 모델이 빠르게 적응해야 함
- 모델을 미세 조정할 수 있어야 함
- 우리 모델은 신경망 기반이라 쉽게 미세조정 할 수 있음
